## データフロー

```
[スクレイピング] 
    ↓ raw HTML / JSON
[ETL + パース]
    ↓ 構造化データ
[PostgreSQL] ──→ [特徴量エンジニアリング]
                     ↓
                [特徴量 DataFrame]
                     ↓
                [XGBoost 学習] ──→ [モデル保存]
                                    ↓
                                [ウォークフォワード検証]
                                    ↓
                                [精度レポート生成]
```

## 主要なエンティティ（予定）

| テーブル | 内容 | 主キー |
|---|---|---|
| `teams` | 24チームのマスタ | `team_id` |
| `players` | 選手マスタ（年齢、ポジション、国籍等） | `player_id` |
| `games` | 試合マスタ + 結果 | `game_id` |
| `box_scores` | 試合ごとのチーム / 選手スタッツ | `game_id, player_id` |
| `rosters` | 試合ごとのロスター（誰が出られるか） | `game_id, team_id, player_id` |
| `injuries` | 欠場情報の履歴 | `player_id, start_date` |
| `predictions` | 予測結果の履歴 | `game_id, predicted_at` |
| `features_cache` | 特徴量キャッシュ（再計算回避） | `game_id, feature_set_version` |

## 主要な設計判断

### 1. モデル分離ではなく「1モデル + CS補正」方式を採用
- 理由: CS のサンプル数が少ない（過去10年で ~150試合弱）
- 実装: レギュラー + CS を共通で学習。CS 用の補正パラメータ（ホームアドバンテージ増強、ELO K値低下等）を適用

### 2. 外国籍選手は直近3年以内のデータに限定
- 理由: Bリーグでは外国籍のシーズン中入れ替えが頻繁
- 実装: 選手ごとの使用可能スタッツ期間を 3年でカット

### 3. 日本人選手は年齢ピーク曲線で重み付け
- 理由: ポジション別にピーク年齢が存在（PG 28-31歳、C 27-31歳 等）
- 実装: `peak_score(age, position)` 関数で 0-1 の重みを算出し、出場時間加重でチーム値に集約

### 4. ウォークフォワード検証は週次再学習
- 理由: 毎試合再学習は計算量過大。月次は粗すぎる
- 実装: 2025-26 シーズンを週単位でスライスし、各週頭で再学習 → その週の試合を予測

### 5. Docker ベースでローカル完結
- 理由: DB 含めて環境依存を排除、どの Mac でも再現可能に
- 実装: postgres コンテナ + app コンテナ。data 永続化は named volume

## セキュリティ / プライバシー方針

- スクレイピング頻度は抑制（Crawl-delay 相当 10秒以上、深夜帯優先）
- User-Agent はアプリ名と連絡手段を明示（紳士的運用）
- 認証情報（DB パスワード、将来的な API キー）は `.env` のみ。`.env` は Git 管理外
- スクレイピング結果データは `data/` 以下に保存、Git 管理外
- モデル・成果物の外部公開は行わない
